SystemUI中添加右侧全局菜单

TV开发过程中，需要用按键弹出一个全局的菜单，来调节一些画面设置，音响效果，所以需要去修改系统添加对按键响应，并弹出菜单界面。
分析：基于以上需求，首先要自定义一个全局按键，并在系统中对这个全局按键的事件监听，监听这个按键事件后，再弹出菜单界面。
问：菜单界面在哪里定义，怎么调用出来？
考虑到这是一个全局的界面，在任何界面都要能调出来，所以将它当SystemUI处理，因此可以添加全局菜单
至于怎么调出来，这里我参考了APP_SWITCH按键响应的过程，也就是Android的后台任务切换键。
下面记录一下开发的过程：

  1. 自定义按键的添加与响应

    • 自定义按键就不多做记录了，之前做自动化测试的时候已经添加过了
    • 按键事件在输入后会经过两次分发，分别是interceptKeyBeforeQueueing()及interceptKeyBeforeDispatching()，interceptKeyBeforeQueueing()先调用，在这个方法中，如果事件处理后返回值为0，则说明这个事件被拦截了。
    • interceptKeyBeforeDispatching()方法将对按键事件做二次拦截，返回值为-1说明事件被拦截，返回值为0则说明事件被放行。
    • 按键事件输入的分发具体过程可以参考：Android输入事件从读取到分发五：事件分发前的拦截过程

  2. 在系统的按键事件中，处理特定按键的位置就在interceptKeyBeforeDispatching()方法中，因此我们对自定义按键的处理也可以放在这个方法中，处理KEYCODE_APP_SWITCH具体代码：

  3. 仿照toggleRecentApps()方法，将打开Menu交给SystemUI去处理，在toggleMenu()方法中，调用了IStatusBarService的toggleMenu()方法，因此下一步就要在IStatusBarService的AIDL文件中声明这个方法。

  4. 实现IStatusBarService接口定义中添加的toggleMenu()方法，在Android App Switch切换流程文章中，分析到IStatusBarService接口的具体实现类是StatusBarManagerService，因此要在StatusBarManagerService类中实现toggleMenu()方法，同样仿照toggleRecentApps()方法，具体实现：

  5. 在StatusBarManagerService中，所有的调用都是通过IStatusBar对象mBar来实现的，而IStatusBar也是一个AIDL的远程接口，因此也需要在IStatusBar的aidl文件中添加toggleMenu()方法的定义，并在它的具体实现类中实现这个方法，这个类就是SystemUI中的CommandQueue，而在CommandQueue中又将具体实现抛给了Callbacks回调接口，再跟踪Callbacks的实现，最终实现是BaseStatusBar，因此就在BaseStatusBar中实现最终的全局菜单。

  6. 最终实现的结果与预期一致，但是遇到一个奇怪的问题，就是在APK中实现的菜单移植到SystemUI中来，出现动画时出现锯齿的问题，经查原来差异在是否开启了硬件加速，这里也就记录一下，关于Android硬件加速，参考：Android setLayerType 硬件加速问题。

    • 另关于如何查看是否开启硬件加速的方法：

      View.isHardwareAccelerated()：如果View附加到一个硬加速的window上就返回true．
      Canvas.isHardwareAccelerated()：如果Canvas被硬加速了就返回true．

// 启动SystemUI
static final void startSystemUi(Context context) {
    Intent intent = new Intent();
    intent.setComponent(new ComponentName("com.android.systemui",
                "com.android.systemui.SystemUIService"));
    //Slog.d(TAG, "Starting service: " + intent);
    context.startServiceAsUser(intent, UserHandle.OWNER);
}

  • 2.启动SystemUIService服务，在SystemUIService的OnCreate()方法中，创建实例：

    /**
 * The classes of the stuff to start.
 */
private final Class<?>[] SERVICES = new Class[] {
        com.android.systemui.recent.Recents.class,
        // SystemBars的全类名
        com.android.systemui.statusbar.SystemBars.class,
        com.android.systemui.usb.StorageNotification.class,
        com.android.systemui.power.PowerUI.class,
        com.android.systemui.media.RingtonePlayer.class,
        com.android.systemui.settings.SettingsUI.class,
    };

/**
 * Hold a reference on the stuff we start.
 */
private final SystemUI[] mServices = new SystemUI[SERVICES.length];
@Override
public void onCreate() {
HashMap<Class<?>, Object> components = new HashMap<Class<?>, Object>();
final int N = SERVICES.length;
for (int i=0; i<N; i++) {
    // 反射创建实例
    Class<?> cl = SERVICES[i];
    Log.d(TAG, "loading: " + cl);
    try {
        mServices[i] = (SystemUI)cl.newInstance();
    } catch (IllegalAccessException ex) {
        throw new RuntimeException(ex);
    } catch (InstantiationException ex) {
        throw new RuntimeException(ex);
    }
    mServices[i].mContext = this;
    mServices[i].mComponents = components;
    Log.d(TAG, "running: " + mServices[i]);
    // 启动start()方法
    mServices[i].start();
}
  • 3.在SystemUIService中创建了SystemBars的实例，并给它的mContext及mComponents赋值，最后调用了它的start()方法，在start()方法中创建一个ServiceMonitor的实例，并将自己的引用传给了ServiceMonitor，最后调用了ServiceMonitor的start()方法。
    @Override
public void start() {
    if (DEBUG) Log.d(TAG, "start");
    mServiceMonitor = new ServiceMonitor(TAG, DEBUG,
            mContext, Settings.Secure.BAR_SERVICE_COMPONENT, this);
    mServiceMonitor.start();  // will call onNoService if no remote service is found
}
  • 4.在ServiceMonitor的start()方法中，做了最重要的一件事就是启动服务:mHandler.sendEmptyMessage(MSG_START_SERVICE)—-> 导致调用startService()
    private void startService() {
    mServiceName = getComponentNameFromSetting();
    if (mDebug) Log.d(mTag, "startService mServiceName=" + mServiceName);
    if (mServiceName == null) {
        mBound = false;
        mCallbacks.onNoService();
    } else {
        long delay = mCallbacks.onServiceStartAttempt();
        mHandler.sendEmptyMessageDelayed(MSG_CONTINUE_START_SERVICE, delay);
    }
}
  • 5.这其中的mCallbacks就是SystemBars的引用，第一次的时候交会回调它的onNoService()方法，在这个方法中只做了一件事件那就是调用了createStatusBarFromConfig()方法：
    private void createStatusBarFromConfig() {
    if (DEBUG) Log.d(TAG, "createStatusBarFromConfig");
    // config_statusBarComponent字符串的内容：com.android.systemui.statusbar.phone.PhoneStatusBar
    final String clsName = mContext.getString(R.string.config_statusBarComponent);
    if (clsName == null || clsName.length() == 0) {
        throw andLog("No status bar component configured", null);
    }
    Class<?> cls = null;
    try {
        cls = mContext.getClassLoader().loadClass(clsName);
    } catch (Throwable t) {
        throw andLog("Error loading status bar component: " + clsName, t);
    }
    try {
        mStatusBar = (BaseStatusBar) cls.newInstance();
    } catch (Throwable t) {
        throw andLog("Error creating status bar component: " + clsName, t);
    }
    mStatusBar.mContext = mContext;
    mStatusBar.mComponents = mComponents;
    mStatusBar.start();
    if (DEBUG) Log.d(TAG, "started " + mStatusBar.getClass().getSimpleName());
}

  • 6.在createStatusBarFromConfig()方法中，R.string.config_statusBarComponent的内容就是com.android.systemui.statusbar.phone.PhoneStatusBar，PhoneStatusBar的全类名的字符串，通过反射创建PhoneStatusBar的实例，并给它的mContext及mComponects赋值，最后调用了它的start()方法。

  • 7.到这里，PhoneStatusBar的实例就创建成功了。

IStatusBarService 的AIDL实现与调用

  • IStatusBarService.aidl文件路经：

    /frameworks/base/core/java/com/android/internal/statusbar/IStatusBarService.aidl

  • 定义的aidl文件会在编译的时候生成IStatusBarService.java文件，生成的路经：

    out/target/common/obj/JAVA_LIBRARIES/framework-base_intermediates/src/core/java/com/android/internal/statusbar/IStatusBarService.java

  • 而空上IStatusBarService的返回是在SystemUI的BaseStatusBar.java文件中：

  • StatusBarService的实现还是在ServiceManager中初使化的，而ServiceManager的实际初使化：

  • 因此StatusBarService的最终实现是StatusBarManagerService.java,再看一下内部实现：

  • 它的内部实现非常简单就是都交给mBar去实现，而这个mBar我在Android App Switch切换流程文章中分析过，它的实际引用是BaseStatusBar，它注册的时候是以CommandQueue的形式注册的，CommandQueue是继承自IStatusBar.Stub，IStatusBar也是一个AIDL的远程调用，它的定义IStatusBar.aidl，与IStatusBarService在同一个路经。

  • 在CommandQueue中定义了一个Callbacks接口，而IStatusBar接口中的定义都是通过Callbacks来调用的。
  • 具体的CommandQueue.Callbacks接口中定义的方法的实现类是BaseStatusBar，所有具体引用是BaseStatusBar
